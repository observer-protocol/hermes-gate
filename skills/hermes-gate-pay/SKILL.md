# hermes-gate-pay

Route all MPP payments through the hermes-gate spend gate. The gate checks your signed SpendMandate before spending, executes the payment itself, and records the spend atomically. Call `gate_pay` instead of `mppx` directly.

## When to use

Whenever you need to pay for an API, data, or service that returns an HTTP 402 Payment Required response on a Lightning/MPP rail. The gate checks your mandate, executes the payment if approved, and returns the result. It fails closed if the gate is unreachable or if amount/currency cannot be determined from the 402 header.

## Prerequisites

The hermes-gate MCP server must be running and wired into your config:

```yaml
mcp_servers:
  hermes-gate:
    command: node
    args: [/path/to/hermes-gate/src/mcp-server.js]
    env:
      HERMES_MANDATE_PATH: /path/to/spend-mandate.json
      HERMES_AGENT_DID: "did:key:z6Mk..."
      HERMES_GATE_HTTP_PORT: 8472
```

`mppx` must be installed and on PATH:

```
npm install -g mppx
mppx account create
```

Run `gate_status()` to confirm the gate is up and your mandate is loaded.

## How to pay

Call `gate_pay` with the 402-protected URL:

```
gate_pay(url="https://api.example.com/resource")
```

Do not call `mppx` directly. The gate probes the 402 URL for amount and currency, evaluates against your mandate (per-transaction ceiling, 24h rolling cap, rail allowlist), and executes the payment if approved.

## Parameters

| param | type | required | description |
|---|---|---|---|
| `url` | string | yes | The 402-protected URL to pay |
| `wallet_id` | string | no | Wallet DID for BIND verification when WalletBindingCredential is configured |

## Response fields

| field | meaning |
|---|---|
| `allowed` | Whether the gate approved and executed the payment |
| `amount` | Amount paid (confirmed if parsed, probed if reconciliation_required) |
| `currency` | Currency of the payment |
| `tx_ref` | Payment hash from mppx (null if not parsed) |
| `reasons` | Denial reasons when allowed=false |
| `reconciliation_required` | true when mppx exited 0 but output did not parse; ledger recorded at probed amount; flag for operator review |

## Procedure

1. When you need to access a 402-protected resource, call `gate_pay(url=<the url>)`.
2. `allowed: true`: payment was made; access the resource.
3. `allowed: false`: spend denied by mandate. Read `reasons` for the specific rule. Do not retry without operator review.
4. `reconciliation_required: true`: payment likely went through but confirmation did not parse. Flag for operator review before retrying.

## What this enforces

The gate checks your signed SpendMandate on every call:

- Per-transaction ceiling: any single spend above the ceiling is denied.
- Rolling 24h cap: cumulative spend is tracked in a local spend ledger; the gate reserves against the cap on evaluate and commits on confirmed payment, preventing concurrent calls from each passing the same headroom.
- Rail allowlist: only configured rails are permitted; anything else is denied.

The mandate is signed by your principal key. The agent cannot rewrite it.

## Why gate_pay instead of mppx

The MPP Agent skill auto-pays on HTTP 402 — the agent calls `mppx` directly without consulting the gate first. `gate_pay` closes that loop: the mandate check and the payment happen in the same step, so the gate always sees the real amount being paid. The `pre_tool_call` interceptor installed by `hermes-gate plugin install` remains active as a backstop for any direct `mppx` invocations.

## Claim scope

gate_pay + this skill is soft-binding: the agent is routed through the gate by instruction. An adversarial agent that constructs a payment call outside the MCP interface is not blocked by this tier. Hard-binding (custody of the transacting key) is the next milestone.
